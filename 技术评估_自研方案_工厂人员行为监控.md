### 1.2 业务与物理环境约束

- **过道场景，无固定工位**：人员处于移动状态，没有“固定工位-固定人员”的绑定逻辑，需支持动态追踪。
- **纯视觉方案（摄像头 + 人）**：现场只有监控摄像头和走动的工人，没有任何 RFID 刷卡、指纹或门禁刷卡等物理辅助。
- **不采集人脸信息**：由于远距离人脸检测极难且涉及隐私，主要通过员工穿戴带有清晰大号码的衣服（如反光马甲）来识别和锁定个人身份。
- **低成本要求**：优先使用普通或稍微高清的监控摄像头，避免高成本的高清人脸抓拍相机。
- **实时性与追溯性**：需要实时判定异常行为（过道静止、低头等）、即时告警，并记录历史告警以供查询。

### 3.3 行为检测算法设计

#### (1) 长时间站立不动（过道异常停留）

```
算法逻辑：
1. 人体检测 + ByteTrack 跟踪，在过道视频流中持续追踪每个人的 track_id 与 bbox（边界框）。
2. 计算每个 track_id 在滑动窗口时间（如 3 分钟 = 4500 帧 @25fps）内的累计位移。
3. 判定规则：如果中心点位移持续低于阈值（如在过道某一区域停留且位移 < 50 像素），则判定为“静止”。
4. 排除误报：若该过道区域存在合理的临时等待点（如饮水机旁、配电箱前等），可通过 ROI（感兴趣区域）配置白名单予以排除；否则一律触发“过道异常静止/摸鱼/身体不适”告警。

关键点：跨帧目标跟踪的稳定性，避免由于光照或微小晃动导致位移超标；针对多人交错遮挡导致 ID 漂移与计时重置，需采用空间静态锚点继承与遮挡状态机等抗干扰机制（详见第十一章）。
难度：★★★☆☆（中，需结合空间锚点与计时继承）
```

#### (2) 长时间低头

```
算法逻辑：
1. PP-TinyPose 提取人体关键点（头部：鼻子、左眼、右眼、左耳、右耳、颈部）
2. 计算头部俯仰角：通过 鼻子-颈部 的垂直夹角
3. 如果俯仰角 > 阈值（如 45°）且持续 > N 秒（如 30 秒）
4. 触发"长时间低头"告警

关键点：摄像头角度对姿态估计影响大，需针对不同安装角度做校准
难度：★★★☆☆（中）
```

#### (3) 摔倒检测

```
算法逻辑：
1. PP-TinyPose 提取人体关键点
2. 计算人体宽高比（bbox）+ 躯干倾斜角 + 头部高度
3. 规则判定：
   - 人体宽高比突然变化（从竖长变横宽）
   - 头部关键点高度骤降（低于髋部关键点）
   - 躯干与地面夹角 < 30°
4. 连续 3~5 帧确认（避免误报）
5. 触发"摔倒"告警

关键点：蹲下/坐下 vs 摔倒的区分，需要时序分析
难度：★★★☆☆（中）
```

#### (4) 多人聚集

```
算法逻辑：
1. 人体检测，统计 ROI 区域内的人数
2. 如果人数 > 阈值（如 3 人）且持续 > N 秒（如 60 秒）
3. 触发"人员聚集"告警

关键点：最简单的检测项，几乎无技术难点
难度：★☆☆☆☆（极低）
```

#### (5) 衣服号码识别（“抓拍门” + MOT 追踪绑定）

```
算法逻辑：
1. 【划定抓拍门（Capture Zone）】：在每个过道摄像头下方或距离摄像头 2~4 米、光线良好、清晰度最高的必经之地区域，手动配置一个虚拟的多边形作为“抓拍门”。
2. 【触发高频 OCR】：当目标（track_id）走入该“抓拍门”时，触发号码 OCR 逻辑：
   - 提取该目标的多帧人体裁剪图像（裁剪框）。
   - 送入训练好的轻量级“衣服号码定位模型（YOLO）”找到号码区域。
   - 对号码区域进行校正与裁切，送入“数字分类/识别模型”中进行推理。
3. 【多帧投票融合】：在抓拍门区域内，对连续 5-10 帧 of 识别结果进行“多数投票（Majority Voting）”。只有在置信度高于阈值（如 90%）且多次识别一致时，才将识别出的数字（如 "08"）与该 `track_id` 绑定（即 `track_id.identity = "08"`）。
4. 【基于跟踪的身份维持】：一旦 `track_id` 绑定了身份，当它离开“抓拍门”向过道深处行走、转身、或被短暂遮挡时，算法不再运行高负荷的 OCR 识别，而是利用多目标跟踪算法（ByteTrack）以及 ReID（行人重识别外观特征）锁死该 `track_id` 的轨迹。
5. 【异常行为关联】：当系统在远端过道检测到该 `track_id` 发生“长时间静止”或“低头”等异常行为时，系统直接拉取其已绑定的 identity 属性（即 "08" 号），生成并推送“员工08号在过道A发生异常行为”的告警。

关键设计参数：
- 衣服规范：衣服的前胸和后背必须印制清晰、巨大、高对比度的数字（详见第八章）。
- ReID 模型支持：在 ByteTrack 算法中加入轻量级 ReID 特征提取器，避免两人交错时导致 ID 跳变、身份发生混淆。

难度：★★★☆☆（中，结合抓拍区和多目标跟踪后，可大幅降低全流程 OCR 识别的难度）
```

## 八、号码衣服纯视觉方案的可行性与核心设计规范

在没有 RFID、没有固定工位，且无法采集人脸的情况下，**“衣服印制号码”是目前性价比最高且唯一可行的身份绑定手段**。虽然号码存在易遮挡、易折皱、远距离模糊等客观技术难点，但我们可以通过“物理规范优化”与“算法分工配合”来彻底解决这些落地痛点。

### 8.1 物理优化规范（从源头降低视觉识别难度）

要保证远距离过道下的识别率，必须对衣服的印制进行标准化规范：

1. **印制位置：前胸 + 后背 双面印制**
   - 避免员工在过道中背对或面对摄像头时产生盲区。确保无论员工朝哪个方向走，至少有一面能完全展现在摄像头视线内。
2. **号码尺寸：超大号设计**
   - 单个数字高度不得低于 **20厘米**，宽度不低于 **10厘米**，笔画粗细不低于 **2.5厘米**。
3. **字体选择：无衬线、防混淆字体**
   - 统一采用 **Impact** 或 **Arial Black** 等粗体无衬线字体，这类字体边缘平整、笔画饱满，有利于 OCR 定位。
   - 针对容易混淆的数字组合（如 0 和 8，1 和 7，5 和 6），在数字下边缘加下划线，或者调整衣服号码的段落间隔。
4. **色彩搭配：极高对比度（高反光）**
   - 推荐使用**“荧光黄/荧光橙反光马甲 + 纯黑大数字”**。黑色大数字在亮色和反光背景的衬托下，即使在光线通道远端，轮廓也极其清晰。
5. **备用方案：安全帽编号**
   - 衣服在弯腰或手抬起时可能起皱，安全帽的形状则是绝对固定的。可以在员工安全帽的左、右、后三侧贴上对应的反射大字号编号，作为衣服号码的辅助视觉输入。

### 8.2 算法分工架构（如何用算法避开“衣服褶皱”和“远距离模糊”）

纯视觉方案成功的关键，在于**不把“衣服号码识别”当成一个全天候、全距离的任务**，而是用 **“抓拍门 + 轨迹跟踪”** 的组合拳：

```
                    ┌─────────────────────────┐
                    │      员工进入过道       │
                    └────────────┬────────────┘
                                 │
                                 ▼
                    ┌─────────────────────────┐
                    │  进入“抓拍门”（近景区） │
                    └────────────┬────────────┘
                                 │
                    ┌────────────┴────────────┐
                    │ 触发多帧OCR + 多数投票   │
                    └────────────┬────────────┘
                                 │
                                 ▼
                    ┌─────────────────────────┐
                    │ 成功绑定：Track ID 01   │
                    │      = 号码 "08"         │
                    └────────────┬────────────┘
                                 │
                                 ▼
                    ┌─────────────────────────┐
                    │  走出抓拍门，深入过道   │
                    │  (此时号码看不清或被挡)  │
                    └────────────┬────────────┘
                                 │
                      ┌──────────┴──────────┐
                      │  依靠 MOT 算法持续  │
                      │  跟踪 Track ID 01   │
                      └──────────┬──────────┘
                                 │
         ┌───────────────────────┴───────────────────────┐
         ▼                                               ▼
┌─────────────────────────┐                     ┌─────────────────────────┐
│     正常走动、离场      │                     │   发生异常（低头/静止） │
└─────────────────────────┘                     └───────────┬─────────────┘
                                                            │
                                                            ▼
                                                ┌─────────────────────────┐
                                                │ 触发告警：              │
                                                │ “员工 08 号异常停留”   │
                                                └─────────────────────────┘
```

1. **局部精准识别（避开远距离）**：
   - 监控范围内 8 米开外全部交给多目标跟踪器（MOT）记录轨迹，不做 OCR 运算。
   - 只有在目标经过摄像头下方 2~4 米的“抓拍门”时，系统提取高质量图像进行 OCR。这能保证获取足够的像素 and 高对比度信息。
2. **时序多数投票（避开衣服褶皱与偶然遮挡）**：
   - 员工经过抓拍区一般需要 1~2 秒，在此期间摄像头会产生 25~50 帧图像。
   - 过滤其中发生大幅度晃动或褶皱严重的帧（通过检测置信度），对剩下 10 帧以上的清晰图像进行 OCR 独立识别。
   - 取识别频率最高的数字作为绑定值（例如有 8 帧识别为 "08"，1 帧识别为 "00"，1 帧识别为 "09"，则最终判定身份为 "08"）。
3. **外观 ReID 辅助关联（避开轨迹短暂丢失）**：
   - 若员工在过道深处被其他工人短暂完全遮挡，导致多目标跟踪 ID 丢失：
   - 当其再次显现时，系统通过 ReID 特征（如衣服颜色比例、身高等）重新匹配之前的轨迹。如果识别概率较高，直接继承之前的 identity。
   - 若 ReID 无法确认，当此人再次走入过道任何一个抓拍门时，会无感触发第二次绑定。

## 九、纯视觉场景下的身份辅助绑定与降噪方案

在老板坚决要求“不采集人脸、无 RFID、只有摄像头和人”的商业约束下，我们可以通过以下两种低成本的纯视觉物理辅助手段，与号码衣服相配合，进一步提升系统的可靠性和抗噪能力：

### 9.1 班组马甲分色管理（视觉初筛）

- **原理**：将不同班组、不同工种或不同岗位的员工，分配不同颜色的反光马甲（例如：装配组穿荧光黄，质检组穿荧光橙，物流组穿荧光绿）。
- **算法配合**：
  1. 目标进入视频流后，首先通过简单的**主颜色提取算法**（甚至利用 YOLO 物体检测模型直接分类“黄色马甲”、“橙色马甲”等），实现人员的粗粒度分类。
  2. 这样做可以将号码的候选空间大大缩小。例如，如果识别到马甲是黄色，且 OCR 模糊识别出数字像 6 或 8，系统可以直接在黄色班组的“号码池”中匹配，大大提升 OCR 纠错和模糊识别的准确率。
- **优点**：视觉表现力强，老板一眼能看出不同班组的分布，且算法实现成本极低。

### 9.2 安全帽贴号（防止褶皱干扰）

- **原理**：安全帽是刚性结构，无论工人怎么转身、弯腰，安全帽上的标贴永远不会发生像衣服那样的褶皱形变。
- **算法配合**：
  - 员工上岗时，在安全帽的前、后、左、右侧粘贴大字号反射贴纸（与衣服号码一致）。
  - 在“抓拍门”算法中，除了裁剪衣服区域外，系统同时检测并裁切安全帽区域。由于帽子形状规则，OCR 对帽子上的数字识别准确率往往高于起皱的衣服。
- **优点**：与号码衣服互为冗余备份。当衣服号码被手臂遮挡时，头顶的安全帽号码依然完全暴露在摄像头的俯视视角下。

### 9.3 汇报话术建议（打消老板对“识别率”的疑虑）

> “老板，您提的‘衣服号码代替人脸识别’这个点子非常切合我们的现场实际！在过道这种无工位、没法刷卡的流动区域，衣服号码确实是最低成本 of 视觉绑定方案。
>
> 之前我们担心距离远了看不清数字，现在我们用算法解决了这个问题：我们在摄像头下方划定一个**‘抓拍门’**，只要员工走过去，系统就能趁离得近、最清楚的时候把号码认出来，并绑定到他的轨迹上。之后他就算走远了、转身了，系统也依然用目标追踪技术认得他。
>
> 为了让这个方案 100% 落地，我们需要做两个配合：第一，衣服正反面都要印大号黑色数字，最好配上高对比的反光马甲；第二，我们在安全帽上也贴上相同的号码。这样双重保险，系统识别率会非常高，既省去了人脸识别的成本和隐私风险，又不需要搞昂贵且难以铺设的 RFID 设备，完全做到了‘只有摄像头和人’就能管好过道秩序。”

## 十一、过道复杂场景：多人聚集定位与强遮挡抗干扰（ID Switch）专项方案

在无固定工位、无 RFID、不采集人脸的纯视觉过道监控中，最核心的两个挑战是：
1. **多人聚集场景下个人的精确定位与行为判定**。
2. **行人交错遮挡导致目标跟踪 ID 飘移（ID Switch），从而导致静止、低头等异常行为的累计计时被重置归零，造成系统漏报**。

为彻底解决上述痛点，我们设计了以下**“多目标空间投影定位与多级抗遮挡轨迹-计时继承算法”**。

### 11.1 多人聚集下的个人分辨与定位（单目逆透视与 BEV 投影）

在过道中，几个人聚集在一起时，由于画面重叠和透视畸变，仅在 2D 像素平面计算距离是极不准确的。为此，我们采用 **“检测（YOLO）+ 追踪（ByteTrack）+ 单目逆透视映射（IPM）”** 的技术路线：

1. **个体分辨（Individual Resolution）**：
   - 采用 **ByteTrack** 追踪算法。即使多人在过道内聚集并发生部分身体遮挡，ByteTrack 也会将低置信度的人体候选框（检测置信度在 0.1 到 0.6 之间）纳入双向数据关联。这极大地防止了聚集时目标因被遮挡而直接丢失。
2. **物理定位（BEV 空间重构）**：
   - **单目逆透视变换（IPM）**：在系统部署时，通过摄像头内参和外参标定，或者手动在过道地面选取 4 个已知物理尺寸的坐标点，计算出从**像素坐标系**到**工厂地面物理坐标系**的单应性矩阵（Homography Matrix）。
   - **BEV（Bird's Eye View，鸟瞰图）投影**：在运行推理时，提取每个人体边界框（BBox）底边的中心点（即工人的双脚踩地位置像素坐标 `[u, v]`）。
   - 通过单应性矩阵，将像素坐标 `[u, v]` 投影为工厂过道的 2D 物理地图坐标 `[X_real, Y_real]`（单位：米）。
   - 这样即可在 BEV 视图下，排除远近透视误差，获得每个人在过道上的精确物理位置，精确定位到个人。

### 11.2 聚集及个人行为检测的定位算法

基于 IPM 映射后的 2D 物理坐标，我们设计了针对聚集和异常行为的鲁棒性检测机制：

1. **多人聚集的快速定位与判定（物理空间聚类）**：
   - **聚集区域识别**：使用实时 **DBSCAN（具有噪声的基于密度的聚类方法）** 算法，对当前帧过道内所有工人的物理坐标 `[X_real, Y_real]` 进行空间聚类。
   - **聚类参数配置**：设置邻域物理距离阈值 $\epsilon = 1.5$ 米，最小聚集人数阈值 $MinPts = 3$ 人。
   - **判定与定位**：若 DBSCAN 聚类出的簇（Cluster）包含的 `track_id` 数量超过阈值，且该簇的质心位置在滑动窗口（如 60 秒）内漂移小于 0.5 米，系统将锁定该区域为“多人聚集区”，输出该聚集簇的物理中心坐标，并向后台推送包含聚集人数和精确物理位置的告警。
2. **聚集人群内个人行为（低头、静止）的精确定位**：
   - 在已分辨出的各个 `track_id` 轨迹中，通过 **PP-TinyPose** 算法提取其独立的骨骼关键点（如鼻子、肩膀、髋部、膝盖）。
   - **低头行为**：在 BBox 内部独立分析关键点，通过计算“鼻子-颈部”的垂直偏角。只有当角速度低于微动阈值（表示头部相对静止）且俯角大于 45° 持续 30 秒时，才判定该 ID 发生低头。
   - **静止行为**：监控该 ID 在物理坐标系 `[X_real, Y_real]` 下的时序轨迹。如果该 ID 的物理坐标在 3 分钟内累计波动半径 $< 0.3$ 米，则判定该个体长时间站立不动。

### 11.3 核心痛点：强遮挡抗干扰与计时继承机制

> [!IMPORTANT]
> **真实痛点场景**：员工 A 已经静止站立了 2 分钟，此时员工 B 从 A 前方走过，遮挡了 A。B 离开后，由于光照或遮挡时间较长，A 被重新检测到时，ByteTrack 赋予了 A 一个新的 `track_id`（发生了 ID Switch）。如果直接按新 ID 重新计算静止时间，A 的 2 分钟历史记录将丢失，导致静止计时重置，严重漏报“长时间静止不动”。

针对此类有人干扰、遮挡的场景，我们设计了**四级防御式抗干扰与计时继承算法**，确保即便 ID 变了，行为计时依然能够完美累加：

#### 1. 第一级防御：卡尔曼滤波惯性预测 + 局部 Re-ID 外观检索

- **卡尔曼滤波（Kalman Filter）**：在目标被完全遮挡期间，ByteTrack 并不立即销毁该 Track 状态，而是利用卡尔曼滤波对其运动学状态（位置、速度、加速度）进行盲测预测。在遮挡发生的前 30 帧（约 1.2 秒）内，如果遮挡物移开，原 ID 将通过高 IoU 关联直接恢复。
- **Re-ID 外观特征融合关联**：当目标丢失超过卡尔曼滤波的生命周期（如 30 帧以上），当人再次出现时，系统提取该新检测框的 **Re-ID 深度外观特征向量**（通过轻量级 ResNet/MobileNet 提取 128 维特征），并计算其与最近 10 秒内丢失的目标特征库之间的余弦相似度。
  - 若余弦相似度 $> 0.85$，则判定为原目标，强制恢复原 `track_id`。
  - **计时恢复**：原 ID 恢复后，直接继承并继续累加之前的静止/低头计时器。

#### 2. 第二级防御：物理空间静态锚点（Static Spatial Anchor）计时继承

如果遮挡时间过长（例如 A 被挡住长达十几秒），或者环境光照突变，导致 Re-ID 提取的特征发生畸变，相似度低于阈值，导致系统**不可避免地为 A 分配了新 ID（ID 漂移发生）**。此时将触发空间锚点继承：

- **创建空间静态锚点**：当任何 `track_id` 被检测到处于“静止状态”且累计静止时间超过某一触发阈值（如 10 秒）时，系统在物理坐标 `[X_real, Y_real]` 处生成一个虚拟的**“静态空间锚点（Static Spatial Anchor）”**。该锚点会记录：
  - 开始静止的时间戳 `start_time`
  - 累计已静止的时间 `accumulated_time`
  - 锚点的物理坐标范围（如以 `[X_real, Y_real]` 为中心，半径 0.8 米的圆）
- **时间继承逻辑**：当原 ID 丢失，随后出现一个新分配的 ID（如 `track_id_new`），如果该新 ID 同时满足以下三个特征：
  1. 它的物理位置落入上述某个**“静态空间锚点”的有效半径内**；
  2. 它的初始运动速度极低（接近 0，判定为非路过行人）；
  3. 它出现的时间与原 ID 在该锚点丢失的时间差在容差窗口内（如 15 秒内）。
- **执行合并**：算法立即判定 `track_id_new` 为原静止工人的延续。**新 ID 直接继承该静态空间锚点的 `start_time` 和历史计时**，合并两者轨迹。
- **效果**：哪怕路人干扰导致原 ID 彻底丢失并变成了新 ID，只要原地的工人没有离开，静止计时就绝不会重置，能够准确触发 3 分钟的异常静止告警。

#### 3. 第三级防御：遮挡状态机（Occlusion State Machine）与计时挂起

为了防止路过行人（非静止人员）的 BBox 压盖到静止人员身上时，导致两者特征混淆，我们引入状态机机制：

- **遮挡相交判定**：计算任意两个 BBox 的交并比（IoU）。如果 $IoU(Box_A, Box_B) > 0.4$，表明两个工人正发生交叉或严重遮挡。
- **计时挂起（Timer Suspend）**：一旦判定发生遮挡，被遮挡的静止工人的行为计时器（静止/低头）将自动进入 **“挂起模式”**：
  - 计时器**暂停走时**，但历史已累计的时间**保存在内存中，不予清空**。
- **状态恢复与释放判定**：当两个 BBox 的 $IoU < 0.1$（说明两人已分开，遮挡结束）后，系统启动 5 秒的“判定窗口”：
  - 若在 5 秒内，该区域的人恢复了静止状态（通过 Re-ID 恢复原 ID 或通过空间锚点继承新 ID），则计时器重新激活，在挂起前的时间基础上**继续累加**。
  - 若 5 秒后，该物理区域内无任何人停留（人员已离开），系统才会正式销毁该静态空间锚点，并清空计时。

```
              ┌────────────────────────────────────────┐
              │          目标 ID_01 判定为静止          │
              │     （创建静态锚点，启动 3分钟计时）     │
              └──────────────────┬─────────────────────┘
                                 │
                                 ▼
              ┌────────────────────────────────────────┐
              │          有路人 ID_02 经过并相交        │
              │         (两目标 IoU > 0.4 发生遮挡)     │
              └──────────────────┬─────────────────────┘
                                 │
```

#### 4. 第四级防御：衣服/安全帽号码异步重绑定

- **重绑定纠错**：如果上述时空关联与 Re-ID 依然发生偏差，当该工人在接下来的站立中，由于稍微转身或抬头，使其衣服号码或安全帽贴号被摄像头捕捉到并成功完成了一次高质量 OCR 识别（例如再次识别为 "08"）。
- **历史回溯合并**：系统会检索当前的活动轨迹库，一旦发现 1 分钟内曾有一个已累计静止 2 分钟的 "08" 号轨迹在相同区域断开，系统会自动将这两个轨迹段进行物理缝合，并把当前的计时时间直接修改为两者之和。这提供了最终的兜底容错保障。

### 12.2 多人聚集场景 — 个人分辨与定位完整算法

```python
def process_frame(frame, homography_matrix, tracked_persons, anchors):
    """
    每帧处理主循环：检测 → 跟踪 → IPM 物理坐标转换 → 聚集检测 → 个体行为判定
    """
    
    # ========== Step 1: 人体检测 + 多目标跟踪 ==========
    detections = yolo_detector.detect(frame)        # PP-YOLOE 检测人体
    tracks = bytetrack.update(detections)            # ByteTrack 更新轨迹
    
    # ========== Step 2: 逐人像素→物理坐标转换（IPM 逆透视） ==========
    all_world_positions = {}
    for track in tracks:
        person = get_or_create_person(track.track_id, tracked_persons)
        
        # 取 BBox 底边中心点作为"脚底"像素坐标
        foot_pixel = (track.bbox.cx, track.bbox.bottom)
        person.pixel_pos = foot_pixel
        
        # 单应性变换：像素坐标 → 物理坐标（米）
        # [X_real, Y_real, 1]^T = H * [u, v, 1]^T
        world_xy = apply_homography(homography_matrix, foot_pixel)
        person.world_pos = world_xy
        
        # 计算物理空间瞬时速度
        person.velocity = calc_velocity(person)
        
        # 提取 ReID 外观特征
        person.reid_feature = reid_model.extract(frame, track.bbox)
        
        all_world_positions[track.track_id] = world_xy
    
    # ========== Step 3: 多人聚集检测（DBSCAN 空间聚类） ==========
    if len(all_world_positions) >= 3:
        positions = np.array(list(all_world_positions.values()))
        ids = list(all_world_positions.keys())
        
        # DBSCAN 聚类：eps=1.5米（邻域半径），min_samples=3（最少3人）
        clustering = DBSCAN(eps=1.5, min_samples=3).fit(positions)
        
        for cluster_label in set(clustering.labels_):
            if cluster_label == -1:  # 噪声点，不构成聚集
                continue
            
            cluster_indices = np.where(clustering.labels_ == cluster_label)[0]
            cluster_ids = [ids[i] for i in cluster_indices]
            cluster_positions = positions[cluster_indices]
            cluster_centroid = cluster_positions.mean(axis=0)
            
            # 聚集告警判定（需持续超过 60 秒）
            check_gathering_alert(
                cluster_ids, cluster_centroid, 
                person_count=len(cluster_ids),
                threshold_seconds=60
            )
    
    # ========== Step 4: 逐人个体行为判定（在聚集中也独立运行） ==========
    for track_id, person in tracked_persons.items():
        if person.world_pos is None:
            continue
        
        # --- 4a: 静止检测（物理坐标位移判定） ---
        displacement = calc_displacement_in_window(
            person, window_seconds=180  # 3 分钟滑动窗口
        )
        if displacement < 0.3:  # 物理空间累计位移 < 0.3 米
            if not person.still_timer.is_running:
                person.still_timer.start(current_time)
                # 静止超过 10 秒，创建空间锚点
                if person.still_timer.accumulated_time > 10:
                    create_or_update_anchor(person, anchors)
            person.still_timer.tick(current_time)
            
            if person.still_timer.accumulated_time >= 180:  # 3 分钟
                trigger_alert("长时间静止", person)
        else:
            person.still_timer.reset()
        
        # --- 4b: 低头检测（姿态关键点 + 静止联动） ---
        keypoints = pose_model.predict(frame, person.pixel_pos)
        head_pitch = calc_head_pitch(keypoints)  # 计算俯仰角
        
        # 关键：仅在"速度极低"（接近静止）时才判定低头
        if head_pitch > 45 and person.velocity < 0.1:  # 静止且低头
            if not person.headdown_timer.is_running:
                person.headdown_timer.start(current_time)
            person.headdown_timer.tick(current_time)
            
            if person.headdown_timer.accumulated_time >= 30:  # 30 秒
                trigger_alert("长时间低头", person)
        else:
            person.headdown_timer.reset()
    
    # ========== Step 5: 遮挡检测 + 状态机管理 ==========
    handle_occlusion(tracked_persons, anchors)
    
    # ========== Step 6: 锚点生命周期管理 ==========
    cleanup_expired_anchors(anchors)
```

#### 遮挡处理与继承核心伪代码

```python
def handle_occlusion(tracked_persons, anchors):
    """遮挡检测 + 状态机 + 计时继承"""
    
    active_ids = list(tracked_persons.keys())
    
    # ========== A: 检测当前帧的遮挡事件 ==========
    for i in range(len(active_ids)):
        for j in range(i + 1, len(active_ids)):
            person_a = tracked_persons[active_ids[i]]
            person_b = tracked_persons[active_ids[j]]
            
            iou = calc_iou(person_a.bbox, person_b.bbox)
            
            if iou > 0.4:
                # 判定谁是"被遮挡的静止者"，谁是"路过的干扰者"
                if person_a.velocity < 0.1 and person_b.velocity > 0.3:
                    # A 是静止者，B 是路过的干扰者
                    suspend_timer(person_a)
                elif person_b.velocity < 0.1 and person_a.velocity > 0.3:
                    # B 是静止者，A 是路过的干扰者
                    suspend_timer(person_b)
                else:
                    # 两者速度都低，可能都是聚集中的人，不触发挂起
                    pass
    
    # ========== B: 检查已丢失 of ID 是否需要锚点继承 ==========
    recently_lost = bytetrack.get_recently_lost_tracks(max_age_seconds=15)
    
    for lost_track in recently_lost:
        # 查找该丢失 ID 是否有对应的活跃锚点
        matching_anchor = find_anchor_by_lost_track(lost_track.track_id, anchors)
        if matching_anchor is None:
            continue
        
        # 在当前帧的新 ID 中，寻找是否有人出现在该锚点范围内
        for track_id, person in tracked_persons.items():
            if person.world_pos is None:
                continue
            
            distance = euclidean_distance(person.world_pos, matching_anchor.world_pos)
            
            # 三重条件判定
            condition_1 = distance < matching_anchor.radius     # 在锚点物理半径内
            condition_2 = person.velocity < 0.1                 # 速度极低（非路过）
            condition_3 = (current_time - matching_anchor.last_associated_time) < matching_anchor.ttl
            
            # 可选的第四重条件：ReID 外观相似度
            reid_score = 0.0
            if matching_anchor.reid_feature is not None and person.reid_feature is not None:
                reid_score = cosine_similarity(person.reid_feature, matching_anchor.reid_feature)
            condition_4 = reid_score > 0.5
            
            if condition_1 and condition_2 and condition_3:
                # ====== 执行计时继承 ======
                # 继承静止计时
                person.still_timer.accumulated_time = matching_anchor.accumulated_time
                person.still_timer.start_time = matching_anchor.start_time
                person.still_timer.is_running = True
                person.still_timer.is_suspended = False
                
                # 继承低头计时（如有）
                if matching_anchor.headdown_accumulated > 0:
                    person.headdown_timer.accumulated_time = matching_anchor.headdown_accumulated
                    person.headdown_timer.is_running = True
                
                # 继承身份绑定
                if matching_anchor.identity:
                    person.identity = matching_anchor.identity
                
                # 更新锚点关联
                matching_anchor.original_track_id = track_id
                matching_anchor.last_associated_time = current_time
                
                break  # 一个锚点只能被一个新 ID 继承


def suspend_timer(person: TrackedPerson):
    """挂起计时器（遮挡期间保留历史时间）"""
    if person.still_timer.is_running and not person.still_timer.is_suspended:
        person.still_timer.is_suspended = True
        person.still_timer.is_running = False
        person.still_timer.suspend_time = current_time
        # 注意：accumulated_time 保持不变，不清零
    
    if person.headdown_timer.is_running and not person.headdown_timer.is_suspended:
        person.headdown_timer.is_suspended = True
        person.headdown_timer.is_running = False


def resume_timer(person: TrackedPerson):
    """恢复计时器（遮挡结束后继续累加）"""
    if person.still_timer.is_suspended:
        person.still_timer.is_suspended = False
        person.still_timer.is_running = True
    
    if person.headdown_timer.is_suspended:
        person.headdown_timer.is_suspended = False
        person.headdown_timer.is_running = True


def cleanup_expired_anchors(anchors, tracked_persons):
    """清理过期锚点"""
    to_remove = []
    for anchor_id, anchor in anchors.items():
        elapsed = current_time - anchor.last_associated_time
        if elapsed > anchor.ttl and anchor.is_active:
            anyone_present = False
            for person in tracked_persons.values():
                if person.world_pos is not None:
                    dist = euclidean_distance(person.world_pos, anchor.world_pos)
                    if dist < anchor.radius and person.velocity < 0.1:
                        anyone_present = True
                        break
            
            if not anyone_present:
                anchor.is_active = False
                to_remove.append(anchor_id)
    
    for aid in to_remove:
        del anchors[aid]
```

### 12.4 边界场景逐一分析与应对策略

> **重要**：以下列出了实际部署中可能遇到的 12 个边界场景，每个场景均附有系统的预期行为和应对策略。开发和测试阶段应逐一覆盖。

#### 场景 1：A 静止 2 分钟，B 快速经过遮挡 0.5 秒后离开

- **系统行为**：遮挡时间极短（< 1.2 秒），卡尔曼滤波惯性预测直接恢复原 ID，计时无中断。
- **防御级别**：第一级防御即可解决。ByteTrack 卡尔曼滤波的默认保活窗口为 30 帧（1.2 秒），0.5 秒的遮挡在保活范围内，ID 不会丢失。

#### 场景 2：A 静止 2 分钟，B 经过遮挡 5 秒后离开，A 获得新 ID

- **系统行为**：遮挡期间计时挂起（保留 2 分钟） → B 离开后触发 ReID 匹配 → 若 ReID 相似度 > 0.85 恢复原 ID → 继续累加。
- **防御级别**：第一 + 三级防御。若 ReID 失败（相似度 < 0.85），触发第二级空间锚点继承：新 ID 在锚点范围内且静止 → 继承 2 分钟计时。

#### 场景 3：A 静止 2 分钟，B 经过遮挡 20 秒后离开，A 获得新 ID 且 ReID 完全失败

- **系统行为**：空间锚点继承：新 ID 满足"在锚点半径内 + 速度极低 + 时间容差 15 秒内" → 强制继承计时。
- **防御级别**：第二级防御兜底。即使 ReID 因长时间遮挡导致特征畸变，只要空间位置匹配且人仍然静止，计时不会丢失。

#### 场景 4：A 静止，B 也在旁边静止（两个人同时站着不动）

- **系统行为**：两人各自维护独立的 track_id 和独立的静止计时器，互不干扰。
- **处理策略**：DBSCAN 聚类将两人归入同一聚集簇，但个体行为判定是按 track_id 独立运行的，不会互相影响。

#### 场景 5：A 静止 2 分钟后离开，B 随后走到 A 原来的位置站住

- **系统行为**：A 离开 → 锚点进入 TTL 倒计时。B 到达时：若 B 的 ReID 特征与锚点存储的 A 的特征差异明显（余弦相似度 < 0.5），系统拒绝继承，B 从 0 开始计时。
- **关键防误判**：空间锚点继承的第四重条件（ReID 二次确认）在此场景下发挥作用。若 B 与 A 穿着截然不同，ReID 会阻止错误继承。若穿着相似（同班组同颜色马甲），则依赖 OCR 号码在后续重绑定中纠错。

#### 场景 6：A 和 B 同时在锚点范围内，都在静止

- **系统行为**：锚点只能被一个新 ID 继承。优先选择距离锚点中心最近且 ReID 最相似的。
- **处理策略**：在 handle_occlusion 中，对候选 ID 按 distance + (1 - reid_score) 综合排序，选择最优匹配者继承。

#### 场景 7：三人聚集在一起，一人低头，突然全部被第四人遮挡

- **系统行为**：三个 track_id 各自的计时器同时挂起。第四人离开后，三人若被重新检测到，各自恢复或通过锚点继承。
- **处理策略**：为每个静止者创建独立的空间锚点，锚点之间通过物理坐标区分（即使相邻，各锚点的圆心和关联的原始 ID 不同）。

#### 场景 8：摄像头画面抖动（风吹/振动），导致所有人的像素坐标都在微晃

- **系统行为**：像素坐标的微晃通过 IPM 映射到物理空间后会被放大或缩小（取决于距离）。
- **处理策略**：在物理坐标上加低通滤波器（如指数移动平均 EMA），平滑 world_pos。静止判定阈值 0.3 米已经有足够的容差吸收相机抖动。

#### 场景 9：同一人反复进出锚点范围（原地来回踱步）

- **系统行为**：踱步的位移会超过静止阈值（0.3 米），静止计时器将被重置。锚点检测到关联者速度 > 0.1，不会被误判为"长时间站立不动"。
- **说明**：这是正确行为。若业务需求是"在某区域长时间逗留（包含踱步）"，可修改为基于物理坐标的逗留半径判定（如在 2 米范围内活动超过 3 分钟）。

#### 场景 10：有人搬运大物件经过，物件被检测为人体误检

- **系统行为**：误检的"人体"会获得 track_id，但其关键点检测得分极低（PP-TinyPose 置信度 < 0.3），不会进入行为判定。
- **处理策略**：在行为判定入口加姿态置信度门控：只有当关键点检测平均置信度 > 0.5 时，才启动低头/静止的行为分析。

#### 场景 11：两个人互换位置

- **系统行为**：如果 ID 未发生跳变，各自计时器随 track_id 正常跟踪移动。如果交换过程中 ID 跳变，锚点会尝试匹配，此时 ReID 外观特征是关键区分依据。
- **处理策略**：若两人穿着差异大（不同颜色马甲/不同号码），ReID 可以正确区分并匹配。若穿着完全相同，最终依赖衣服/安全帽号码 of 异步 OCR 重绑定（第四级防御）。

#### 场景 12：夜班光线极暗，所有人的 ReID 特征都退化

- **系统行为**：ReID 相似度普遍偏低，无法有效区分个体。
- **处理策略**：降级为纯空间锚点模式（降低 ReID 阈值权重）。同时建议在过道关键位置增加补光灯，这是物理层面的根本解决方案。

### 12.6 小结：纯视觉过道场景的技术可行性结论

经过第十一章方案设计和本章的算法伪代码细化，针对用户提出的三个核心问题，我们的结论如下：

| 核心问题 | 解决方案 | 可行性评级 |
|---------|---------|----------|
| **多人聚集时如何分辨个人？** | ByteTrack 多目标跟踪为每人分配独立 ID + PP-TinyPose 为每人独立提取关键点 + IPM 逆透视物理坐标定位 | 高（开源方案成熟） |
| **没有固定工位/RFID，如何定位到个人？** | 单目逆透视映射（Homography）将像素坐标转换为过道物理坐标 + DBSCAN 聚类检测聚集 + 衣服号码 OCR 抓拍绑定身份 | 高（标定一次即可长期使用） |
| **有人干扰遮挡导致 ID 漂移、计时重置？** | 四级防御体系：卡尔曼滤波保活 → ReID 外观恢复 → 空间锚点计时继承 → 遮挡状态机计时挂起/恢复 → OCR 异步重绑定兜底 | 高（多级冗余保障计时不丢失） |

> **开发建议**：第一阶段 MVP 先实现"卡尔曼滤波保活 + 空间锚点继承"（第一、二级防御），这两级已可覆盖 90% 以上的遮挡干扰场景。ReID 外观匹配和遮挡状态机可在第二阶段迭代优化时加入。
